# Tender Triage Reseller MVP

## 1. Целевая аудитория

Тендерные компании и реселлеры, которые ежедневно просматривают много закупок и должны быстро принимать решение: GO / NO-GO / NEEDS REVIEW.

## 2. Боль

Менеджеры вручную просматривают десятки закупок, скачивают документы, ищут спецификации, сроки, обеспечение, ограничения, сертификаты, признаки "только производитель", аналоги и риски. Нужно быстро понять, стоит ли брать закупку в работу.

## 3. Основной сценарий

1. Пользователь открывает `/demo/tender-agent`.
2. Видит основной режим "Быстрый разбор закупки" (search-first).
3. Вводит ключевое слово (кабель, светильник, выключатель и т.д.).
4. При желании задаёт фильтры: закон, регион, НМЦК, дата публикации.
5. Система выполняет поиск.
6. UI показывает количество найденных результатов и список закупок.
7. Система автоматически выбирает самую свежую закупку по дате публикации.
8. UI явно показывает: "Для экономии времени демо анализируется одна закупка — самая свежая из результатов поиска".
9. Система анализирует выбранную закупку.
10. UI показывает отчёт с решением GO / NO-GO / NEEDS REVIEW / LOW PRIORITY.

## 4. Что входит в MVP

- Поиск закупок по ключевому слову с фильтрами
- Отображение количества найденных результатов
- Автоматический выбор самой свежей закупки
- Анализ одной закупки за демо-цикл
- Скоринг 0–100 по 6 критериям
- Решение: GO / NO-GO / NEEDS REVIEW / LOW PRIORITY
- Стоп-факторы с severity: info / warning / critical
- Причины решения
- Рекомендация менеджеру
- Явная маркировка fallback-режима, если поиск недоступен
- Честное отображение ограничения по количеству (total_results)

## 5. Что не входит в MVP

- Автоматическая подача заявки
- Участие в торгах
- ЭЦП
- Финальный расчёт маржи
- Замена менеджера
- Загрузка и анализ произвольных документов (доступно как вторичный режим)
- Анализ по номеру закупки (доступно как вторичный режим)

## 6. Search-first flow

```
search_tenders(query, filters)
    -> search_summary
    -> select_freshest_tender(results)
    -> fetch_tender_details(tender_id)
    -> analyze_tender(tender_details)
    -> render_reseller_triage_report()
```

- `search_tenders` — поиск по demo_local или публичному ЕИС
- `select_freshest_tender` — выбор самой свежей по publication_date DESC
- `fetch_tender_details` — получение карточки закупки
- `analyze_tender` — скоринг и выявление стоп-факторов
- `render_reseller_triage_report` — формирование отчёта

## 7. Decision model

| Решение | Диапазон скоринга | Условия |
|---------|-------------------|---------|
| GO | 75–100 | Нет critical стоп-факторов |
| NEEDS REVIEW | 55–74 | Недостаточно данных или неопределённость |
| LOW PRIORITY | 35–54 | Низкая привлекательность |
| NO_GO | 0–34 | Critical стоп-факторы или низкий скоринг |

Если есть critical stop_factor, решение не может быть GO.
Если не удалось извлечь позиции поставки (no_line_items), решение не может быть GO — возвращается NEEDS REVIEW.
Если данных недостаточно — NEEDS REVIEW.

## 8. Scoring rubric

| Компонент | Вес | Описание |
|-----------|-----|----------|
| Товарное соответствие | 25 | Насколько закупка соответствует товарному профилю |
| Понятность спецификации | 15 | Наличие ТЗ, спецификации, line items |
| Коммерческий потенциал | 20 | НМЦК и привлекательность |
| Сроки и логистика | 15 | Достаточность срока подачи и поставки |
| Документы и допуски | 15 | Доступность и качество документации |
| Контрактные риски | 10 | Типовые риски |

## 9. Stop factors

- `license_sro_required` — лицензия / СРО / специальный допуск
- `manufacturer_only` — признаки "только производитель"
- `short_deadline` — короткий срок до подачи
- `security_required` — обеспечение заявки / контракта
- `smp_restriction` — ограничения по СМП/СОНКО
- `national_regime` — национальный режим / ограничения происхождения
- `certificates_required` — сертификаты / декларации / паспорта качества
- `analogs_unclear` — запрет или неясность аналогов
- `no_line_items` — не удалось извлечь позиции поставки

## 10. Output schema

```python
class ResellerTriageReport(BaseModel):
    decision_label: ResellerDecisionLabel  # GO / NO_GO / NEEDS_REVIEW / LOW_PRIORITY
    decision_score: int  # 0-100
    decision_reasons: list[str]
    stop_factors: list[StopFactor]
    reseller_summary: str
    manager_recommendation: str
    analysis_limit_notice: str
    is_fallback: bool
    fallback_label: str | None

class StopFactor(BaseModel):
    code: str
    title: str
    severity: StopFactorSeverity  # info / warning / critical
    evidence: str
    source: str
```

## 11. Acceptance criteria

1. `/demo/tender-agent` открывается без ошибок.
2. Основной сценарий страницы — поиск по ключевому слову.
3. На странице есть поле ключевого слова и фильтры.
4. Пользователь может запустить поиск закупок.
5. UI показывает количество найденных результатов.
6. UI показывает список найденных закупок (минимум 5, если доступно).
7. Система выбирает самую свежую закупку по дате публикации.
8. Самая свежая закупка явно помечена как выбранная для анализа.
9. За один демо-цикл анализируется только одна закупка.
10. В UI есть явная пометка об анализе одной закупки.
11. Отчёт содержит GO / NO-GO / NEEDS REVIEW / LOW PRIORITY.
12. Отчёт содержит скоринг 0–100.
13. Отчёт содержит причины решения.
14. Отчёт содержит стоп-факторы.
15. Если позиции не извлечены, система не падает, а выставляет NEEDS_REVIEW.
16. Если поиск ничего не нашёл, UI показывает сообщение.
17. Если ЕИС/SOAP недоступен, fallback явно маркируется.
18. Старые режимы (анализ по номеру, загрузка документов) сохранены.
19. Все тесты проходят.

## 12. Demo limitations

- Анализируется только одна (самая свежая) закупка за демо-цикл
- Коммерческий потенциал предварительный
- Для финального решения нужны закупочные цены, логистика и условия поставщиков
- MVP не обещает автоподачу, участие в торгах, ЭЦП
- Решение всегда требует подтверждения человека
- При недоступности источника используется явно маркированный fallback

## 13. Manual demo script (5–7 минут)

### Подготовка

1. Откройте `http://localhost:8000/demo/tender-agent`.
2. Убедитесь, что страница загрузилась, заголовок «Тендерный агент: демонстрация работы».
3. Первая вкладка — «Быстрый разбор закупки» (основной сценарий).

### Сценарий 1: успешный поиск и GO

1. В поле «Ключевое слово» уже предзаполнено «кабель».
2. Нажмите «Найти и разобрать свежую закупку».
3. **Что видит пользователь:**
   - Сообщение «Поиск закупок…» (спиннер).
   - Затем блок с количеством: «Найдено закупок: N».
   - Список результатов с карточками (НМЦК, дата, дедлайн, регион).
   - У одной карточки бейдж «Выбрана для анализа».
4. Ниже — отчёт с решением (GO, NO-GO, NEEDS REVIEW, LOW PRIORITY), скорингом, причинами, стоп-факторами и рекомендацией.
5. **Как объяснять клиенту ограничение одной закупки:**
   - «Для быстрой демонстрации мы анализируем одну самую свежую закупку. В продукте вы сможете анализировать любую закупку из результатов».

### Сценарий 2: другой запрос

1. Очистите поле и введите «светильник» или «автоматический выключатель».
2. Нажмите кнопку поиска.
3. Проверьте, что страница не падает, результаты отображаются, отчёт строится.

### Сценарий 3: поиск без результатов

1. Введите «zzzzzzzz» (заведомо несуществующий запрос).
2. Нажмите кнопку.
3. **Что видит пользователь:**
   - «Ничего не найдено. По вашему запросу закупки не найдены. Попробуйте изменить ключевое слово или фильтры.»
   - Примеры запросов: кабель, светильник, автоматический выключатель, электротехническая продукция.

### Сценарий 4: fallback при недоступном источнике

1. Этот сценарий воспроизводится, когда публичный ЕИС недоступен (сеть отключена).
2. Выполните поиск через public_eis_html_44fz (например, через API: `POST /api/demo/tender-agent/reseller/search-and-triage` с `source: "public_eis_html_44fz"`).
3. UI покажет уведомление:
   - «Источник поиска временно недоступен. Показан демонстрационный набор данных.»
4. **Как объяснять клиенту:**
   - «Если официальный источник (ЕИС) недоступен, мы честно показываем это и предлагаем демо-данные вместо того, чтобы притворяться, что поиск работает. В продукте будет live-статус источника».

### Сценарий 5: вторичные режимы

1. Переключитесь на вкладку «Анализ по номеру».
2. Покажите, что можно получить документацию по реестровому номеру.
3. Переключитесь на «Загрузка документов».
4. Покажите ручную загрузку файлов для полного pipeline.
5. **Как объяснять:**
   - «Это продвинутые сценарии для глубокого анализа. Основной быстрый сценарий — первая вкладка».

### Что НЕ показывать на демо

- Технические названия: runtime, adapter, line_items, fallback_deterministic, .md
- JSON-ответы API
- Ошибки в консоли браузера (developer tools)
- Код и архитектуру (если клиент не спросит)
- Синтетические данные как реальные закупки
- demo_local (даже если используется как dev mode)

## 14. Customer live mode vs synthetic dev mode

### Режимы

**Customer mode (по умолчанию):**
- Использует только реальный поиск закупок (`public_eis_html_44fz`)
- `customer_mode = true` (значение по умолчанию)
- `source_type = "live"` при успехе, `"unknown"` при ошибке
- `synthetic_used = false`
- Если live search недоступен → `search_unavailable = true`, анализ не запускается
- Синтетические данные не используются, fallback на демо-набор запрещён
- UI показывает: «Источник: реальный поиск закупок»

**Dev mode:**
- Разрешены синтетические данные (`demo_local`)
- `customer_mode = false`
- `source_type = "demo"` при использовании `demo_local`
- `synthetic_used = true`
- Включается через query-параметр `?dev=1` на странице UI
- UI показывает красный баннер: «Режим разработки: синтетические данные»

### Поведение search-and-triage

1. `ResellerSearchRequest.customer_mode` определяет режим (по умолчанию `true`)
2. В customer mode параметр `source` игнорируется — всегда используется live search
3. Если live search (ЕИС) недоступен:
   - `status = "search_unavailable"`
   - В теле ответа HTTP 200 с `search.search_unavailable = true`
   - UI показывает ошибку: «Реальный поиск закупок сейчас недоступен.»
   - Анализ не запускается, синтетические данные не подставляются
4. В dev mode `source` учитывается, разрешён fallback на демо-набор

### Что маркирует UI

| Элемент | Customer mode (live) | Customer mode (unavailable) | Dev mode |
|---------|---------------------|----------------------------|----------|
| Бейдж источника | «Реальный поиск» (зелёный) | Не показывается | «Демо-набор» (жёлтый) |
| Баннер | «Источник: реальный поиск закупок» | «Поиск недоступен» (красный) | «Режим разработки: синтетические данные» (красный) |
| Количество | «Найдено минимум N результатов» | Не показывается | «Найдено закупок: N» |
| Анализ | Запускается | Блокирован | Запускается |

### Что говорить клиенту

- **Если live search успешен:**
  «Поиск выполняется по публичным данным ЕИС — это настоящие закупки, доступные для чтения.»
- **Если live search недоступен:**
  «Реальный поиск закупок сейчас недоступен. Проверьте подключение к интернету или повторите позже.»
- **Про одну анализируемую закупку:**
  «Для экономии времени мы показываем разбор самой свежей закупки. В продукте вы сможете анализировать любую закупку из выдачи.»

### Что нельзя делать

- Использовать синтетические данные в customer mode
- Показывать demo result как live procurement
- Говорить клиенту, что synthetic result — это реальная закупка
- Подменять недоступный live search демо-набором в customer mode
