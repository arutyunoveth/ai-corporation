# Ограничения демо

## Что важно понимать

- анализ не является юридическим заключением;
- итоговый вывод требует проверки человеком;
- качество результата зависит от полноты и качества документов закупки.

## Технические ограничения текущего MVP

- in-process jobs не переживают restart backend;
- `detailed` mode сильнее зависит от железа, чем `fast`;
- локальный GGUF path привязан к Mac mini runtime;
- DOCX/PDF форматирование пока MVP-level;
- PDF зависит от доступного локального шрифта с кириллицей;
- export files — runtime artifacts в `data/rag/exports/`;
- orphaned history rows могут вернуть `404`, если исходный saved report file удалён.

## Ограничения по данным и документам

- качество анализа зависит от того, насколько хорошо извлечён текст из документов;
- не все типы закупок одинаково покрыты;
- сканы и сложные документы могут требовать отдельной доработки;
- отсутствие релевантных документов ухудшает качество ответа и `sources_count`.

## Как объяснять ограничения клиенту

Говорить так:

- "Это инструмент для ускорения первичного анализа."
- "Он формирует черновик аналитического отчёта."
- "Источники сохраняются для последующей проверки."
- "Результат помогает специалисту быстрее дойти до решения, а не подменяет его."

Не говорить так:

- "Система полностью заменяет тендерного специалиста."
- "Система гарантирует решение участвовать или не участвовать."
- "Это уже production-ready SaaS для любого сценария."

## Почему это всё равно ценно

- локальный контур;
- прозрачные источники;
- повторяемый запуск;
- история анализов;
- готовый DOCX/PDF для дальнейшего обсуждения.
