# Пошаговый Прогон Пилота С Заказчиком

## Зачем нужен этот документ

Это короткий сценарий живого прогона рядом с заказчиком на одном реальном
тендере.

Логика показа простая:

1. показываем структуру папок
2. показываем входные документы
3. делаем первый прогон
4. показываем требования, риски и RFQ
5. если есть КП, делаем второй прогон
6. показываем сравнение, экономику и итоговый отчет

## Что подготовить до встречи

1. Создать папку прогона:

```bash
mkdir -p local_pilot_runs/<operator_label>/<tender_label>/{01_raw_docs,02_extracted_text,03_supplier_search,04_tkp,05_system_output,06_partner_export,07_feedback}
```

2. Положить оригиналы документов в `01_raw_docs/`.

3. Подготовить три обязательных файла в `02_extracted_text/`:

- `notice.txt`
- `technical_spec.txt`
- `contract_draft.txt`

4. По возможности подготовить `local_pilot_runs/<operator_label>/operator_profile.md`.

5. Если хотите показать вторую половину сценария, заранее положить 2-3 файла
КП в `04_tkp/`.

6. Проверить окружение:

```bash
.venv/bin/python --version
.venv/bin/python scripts/run_tender_operator_pilot.py --help
```

## Как проводить встречу

## Шаг 1. Сразу обозначить границы

Скажите заказчику:

`Сейчас мы показываем управляемый ручной процесс. Система не рассылает письма поставщикам, не подает заявку и не принимает финальное решение вместо человека.`

## Шаг 2. Показать структуру папок

Откройте папку тендера и покажите:

- `01_raw_docs/` — оригиналы документов
- `02_extracted_text/` — тексты для анализа
- `03_supplier_search/` — ручные заметки по поставщикам
- `04_tkp/` — коммерческие предложения поставщиков
- `05_system_output/` — внутренние результаты системы
- `06_partner_export/` — клиентский отчет

Что сказать:

`Вся работа идет от конкретного тендера в локальной структуре папок. Мы не работаем по устному описанию, а опираемся на реальные документы.`

## Шаг 3. Показать три обязательных текстовых файла

Откройте `02_extracted_text/` и покажите:

- `notice.txt`
- `technical_spec.txt`
- `contract_draft.txt`

Что сказать:

`Для честного анализа нам нужны текст извещения, текст ТЗ и проект договора. Это минимальный комплект для первого прохода.`

## Шаг 4. Сделать первый прогон

Запустите:

```bash
.venv/bin/python scripts/run_tender_operator_pilot.py \
  --operator-id <operator_label> \
  --tender-dir local_pilot_runs/<operator_label>/<tender_label> \
  --provider stub \
  --output-dir local_pilot_runs/<operator_label>/<tender_label>/05_system_output
```

Пример:

```bash
.venv/bin/python scripts/run_tender_operator_pilot.py \
  --operator-id tender_operator_001 \
  --tender-dir local_pilot_runs/tender_operator_001/tender_001 \
  --provider stub \
  --output-dir local_pilot_runs/tender_operator_001/tender_001/05_system_output
```

Что сказать во время запуска:

`На первом проходе система собирает структуру требований, выделяет договорные риски, готовит вопросы поставщикам и черновик RFQ. Если КП еще нет, экономика не считается притворно.`

## Шаг 5. Показать результат первого прогона

Откройте по очереди:

1. `05_system_output/requirements.json`
2. `05_system_output/calibrated_contract_risk_memo.md`
3. `05_system_output/supplier_questions.json`
4. `05_system_output/rfq_request_draft.md`
5. `05_system_output/run_summary.json`

Что показать:

1. В `requirements.json` покажите, что требования разложены по структуре.
2. В `calibrated_contract_risk_memo.md` покажите три уровня риска:
   стандартный, материальный, deal-breaker.
3. В `supplier_questions.json` покажите, что система собрала список вопросов
   для поставщиков.
4. В `rfq_request_draft.md` покажите готовый черновик запроса КП.
5. В `run_summary.json` покажите поле `pilot_status`.

Если `04_tkp/` пустая, ожидаемый статус:

- `rfq_ready_collect_tkp`

## Шаг 6. Честно объяснить точку остановки

Если КП еще нет, скажите:

`Вот здесь и есть честная пауза процесса. Дальше оператор вручную отправляет RFQ поставщикам и ждет реальные КП. Без реальных входных предложений система не должна выдумывать сравнение и экономику.`

Если у вас встреча только на первый проход, на этом можно закончить.

## Шаг 7. Показать ручной контур поставщиков

Если заполнен файл `03_supplier_search/supplier_candidates.md`, откройте его.

Что сказать:

`Поиск поставщиков и коммуникация остаются под контролем человека. Система помогает не потерять требования и собрать решение в структуру, но не заменяет оператора в рынке.`

## Шаг 8. Показать файлы КП

Если хотите продолжить во вторую часть, откройте `04_tkp/` и покажите файлы
КП.

Что сказать:

`Каждый файл здесь — это отдельное коммерческое предложение поставщика. После их загрузки можно делать второй проход.`

## Шаг 9. Сделать второй прогон

Запустите ту же команду еще раз:

```bash
.venv/bin/python scripts/run_tender_operator_pilot.py \
  --operator-id <operator_label> \
  --tender-dir local_pilot_runs/<operator_label>/<tender_label> \
  --provider stub \
  --output-dir local_pilot_runs/<operator_label>/<tender_label>/05_system_output
```

Что сказать:

`Теперь у нас есть входные коммерческие предложения, поэтому система может перейти ко второй части: сравнению КП, экономике и предварительной рекомендации.`

## Шаг 10. Показать результат второго прогона

Откройте:

1. `05_system_output/tkp_comparison.json`
2. `05_system_output/economics_summary.json`
3. `05_system_output/bid_decision_recommendation.md`
4. `05_system_output/run_summary.json`

Что показать:

1. В `tkp_comparison.json` покажите, что все КП собраны в одно сравнение.
2. В `economics_summary.json` покажите экономический срез для проверки
   оператором.
3. В `bid_decision_recommendation.md` подчеркните, что это предварительная
   рекомендация.
4. В `run_summary.json` покажите статус `tkp_received_economics_ready`.

## Шаг 11. Показать клиентский отчет

Откройте:

- `06_partner_export/operator_report.md`

Что сказать:

`Это уже внешний отчет для заказчика. Он отделен от внутренних служебных файлов, чтобы наружу не уходили внутренние заметки и чувствительные детали.`

Если нужно, дополнительно откройте:

- `06_partner_export/export_summary.json`

И покажите:

- какие секции включены
- какие секции скрыты
- нет ли блокирующих секций

## Шаг 12. Правильно завершить встречу

Фраза для завершения:

`На одном реальном тендере мы сейчас показали весь контур: от документов и требований до RFQ, а после добавления КП — до сравнения, экономики и предварительной рекомендации. При этом система не пишет поставщикам сама, не подает заявку и не заменяет финальное решение человека.`

## Короткий чеклист перед встречей

- Папка тендера создана
- `notice.txt` готов
- `technical_spec.txt` готов
- `contract_draft.txt` готов
- Команда первого прогона готова
- Файлы КП подготовлены, если нужен второй проход
- `operator_report.md` проверен перед внешним показом

## Короткая версия демо на 10 минут

1. Показать структуру папок.
2. Показать три входных текстовых файла.
3. Сделать первый прогон.
4. Открыть `requirements.json`.
5. Открыть `calibrated_contract_risk_memo.md`.
6. Открыть `rfq_request_draft.md`.
7. Объяснить ручную паузу на сбор КП.
8. Если есть КП, сделать второй прогон.
9. Открыть `tkp_comparison.json`.
10. Открыть `bid_decision_recommendation.md`.
11. Открыть `operator_report.md`.

## Важные оговорки

- Не показывайте на экране сырые файлы из `01_raw_docs/`, если там есть чувствительные данные.
- Не отправляйте заказчику `internal_operator_analysis.md`.
- Не обещайте автоматическую подачу заявки, автоматическую переписку или автономное решение.
- Позиционируйте результат как operator-assisted workflow, а не autonomous bidding.
