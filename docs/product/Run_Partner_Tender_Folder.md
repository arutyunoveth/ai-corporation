# Run Partner Tender Folder

## Purpose

This runbook explains how to use `scripts/run_partner_tender_folder.py` to process a real partner tender from a local folder.

## Prerequisites

- Repository is cloned and dependencies installed.
- Virtual environment is activated.
- You have read:
  - `Local_Pilot_Data_Handling_Policy.md`
  - `Restricted_Paid_Pilot_Operations_Runbook.md`
  - `Redaction_Checklist.md`

## Step 1: Create Local Folder

Create the folder structure for the partner:

```bash
mkdir -p local_pilot_runs/<partner_label>/<tender_label>/{01_raw_docs,02_extracted_text,03_operator_notes,04_system_output,05_partner_export,06_feedback}
```

## Step 2: Place Documents

1. Download tender documents from the partner (email, portal, etc.).
2. Save original documents (PDF, DOCX, etc.) to `01_raw_docs/`.
3. Create plain text extracts in `02_extracted_text/` with these required files:
   - `notice.txt` — The tender notice / announcement text.
   - `technical_spec.txt` — The technical specification text.
   - `contract_draft.txt` — The contract draft text.

## Step 3: (Optional) Create Partner Profile

Create `local_pilot_runs/<partner_label>/partner_profile.md` using the `Partner_Profile_Template.md`. This file is optional but recommended.

## Step 4: Run the Command

```bash
.venv/bin/python scripts/run_partner_tender_folder.py \
  --partner-id partner_001 \
  --tender-dir local_pilot_runs/partner_001/tender_001 \
  --provider stub \
  --output-dir local_pilot_runs/partner_001/tender_001/04_system_output
```

### Arguments

| Argument | Required | Default | Description |
|---|---|---|---|
| `--partner-id` | Yes | — | Partner identifier (used for workspace creation) |
| `--tender-dir` | Yes | — | Path to the tender folder containing `02_extracted_text/` |
| `--provider` | Yes | — | Analysis provider: `stub` (always works) or `llm` (requires DB + API key) |
| `--output-dir` | No | `<tender-dir>/04_system_output` | Where to write system output files |

### Provider Choices

- **`stub`** (recommended for testing): Uses canned synthetic analysis. No external dependencies.
- **`llm`**: Attempts to use the controlled LLM pre-bid analysis module. Requires:
  - PostgreSQL database running with Alembic migrations applied.
  - `AI_CORP_OPENAI_API_KEY` environment variable set.
  - Falls back gracefully with a warning if unavailable.

## Step 5: Review Outputs

After the command completes, check these outputs:

| File | Location | Content | Can Send to Partner? |
|---|---|---|---|
| `run_summary.json` | `04_system_output/` | JSON summary of the run (workspace, records, package IDs) | No (internal) |
| `internal_analysis.md` | `04_system_output/` | Full internal analysis with requirements, risks, recommendation | No (internal — may include operator notes) |
| `partner_report.md` | `05_partner_export/` | Partner-facing redacted report | Yes (after human review) |
| `export_summary.json` | `05_partner_export/` | Export package metadata (included/redacted/blocked sections) | No (internal) |

## Step 6: Manual Review and Delivery

1. Review `partner_report.md` — verify no restricted or internal data is exposed.
2. If the report is clean, deliver it to the partner via your external channel (email, secure portal).
3. Record the delivery in the system:
   - The script already called `approve_for_delivery()` and `mark_delivered_manually()`.
   - This is recorded in `run_summary.json`.

## Step 7: Collect Feedback

1. Use `Partner_Feedback_Template.md` to collect partner feedback.
2. Save notes to `06_feedback/`.
3. Record feedback and outcome using the `pilot_feedback` module or manually.

## Important Rules

- **Never commit raw documents.** `01_raw_docs/` and `03_operator_notes/` are gitignored.
- **Only `05_partner_export/partner_report.md` is partner-ready.**
- **All other output files are for internal use only.**
- **Do not send `run_summary.json` or `internal_analysis.md` to the partner.**
- **Human review is required before every delivery.**
- **No automated external communication is generated by this script.**
